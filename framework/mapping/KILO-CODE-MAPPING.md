# Kilo Code Mapping

> Kilo Code 作为 IDE-centric Agent 的映射参考。

---

## 1. 定位

**Kilo Code 是垂直场景优化样本，不是通用框架。**

- 基于 OpenClaw 构建，但做了 IDE 场景的深度裁剪
- 核心假设：用户是开发者，场景是编码，界面是 IDE
- FAO 需要区分：通用框架 vs 垂直优化的边界在哪里

---

## 2. 与 FAO 核心差异

| 维度 | FAO 通用框架 | Kilo Code |
|------|-------------|-----------|
| **假设的用户** | 异质主体（人 + agent） | 单一主体（开发者） |
| **假设的场景** | 开放任务路由 | 封闭编码场景 |
| **界面形态** | 多平台（CLI/IM/Web） | IDE 插件优先 |
| **Agent 模式** | 单一模式，靠契约收窄 | 6 种预设模式（Code/Architect/Debug/Ask/Custom） |
| **责任锚定** | 显式人类承担 | IDE 内置撤销/历史兜底 |

---

## 3. 关键器官映射

### Multi-Mode 系统
Kilo Code 的 6 种模式是**场景预分类器**，不是运行时动态收窄：
- Code：编码执行
- Architect：架构设计
- Debug：故障排查
- Ask：问答咨询
- Custom：用户自定义

**与 FAO 差异**：FAO 的收窄靠契约检查，Kilo 的收窄靠模式选择。前者是治理层，后者是交互层。

### Cloud Agents
Kilo 的 Cloud Agents 是**能力外包**，不是责任外包：
- 重任务扔到云端执行
- 但结果返回 IDE 后，责任仍由用户承担
- 无显式 human checkpoint 协议

---

## 4. 不可映射之处

- **无组织路由概念**：Kilo 假设单一开发者与单一 agent 的封闭关系，无多主体协作
- **无显式责任契约**：靠 IDE 的撤销/历史功能隐性兜底，无 FAO 式的边界声明
- **无判断-纠错分离**：模式切换是交互选择，不是治理判断

---

## 5. 对 FAO 的有限启发

1. **场景预分类的价值**：6 种模式说明编码任务可被粗分类，FAO 的 domain packs 可参考
2. **IDE 作为约束界面**：IDE 的物理边界（diff 视图、撤销按钮）本身就是一种轻量化 governance
3. **Cloud/Local 分工**：但 FAO 需要显式定义 cloud 执行后的责任归属，Kilo 对此是隐性的

---

## 6. 结论

Kilo Code 是 OpenClaw 生态内的垂直优化案例，证明同一 runtime 可支撑不同场景的形态。但它不能回答 FAO 的核心问题：当多个异质主体协作时，责任如何锚定。
