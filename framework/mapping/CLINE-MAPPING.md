# Cline Mapping

> Cline 作为 Human-in-the-Loop Agent 的映射参考。

---

## 1. 定位

**Cline 是安全优先的妥协方案。**

- 每步操作需人工确认，牺牲效率换可控性
- VS Code 插件形态， IDE 作为天然约束边界
- FAO 需要分析：人工确认是治理还是交互成本？

---

## 2. 与 FAO 核心差异

| 维度 | FAO 通用框架 | Cline |
|------|-------------|-------|
| **安全策略** | 约束前置（契约收窄） | 确认后置（人工兜底） |
| **效率假设** | 正常流转应高效 | 默认低效，换取安全 |
| **界面约束** | 文件化规则 | IDE diff 视图 |
| **权限粒度** | 角色契约预定义 | 单步人工审批 |
| **可复用性** | 约束可复用 | 每次重新确认 |

---

## 3. 关键器官映射

### Human-in-the-Loop
Cline 的每步确认是**交互层的安全网**：
- 文件修改前展示 diff
- 终端命令执行前等待点击
- 浏览器操作可中断

**与 FAO 差异**：FAO 的收窄在任务准入阶段完成，Cline 的收窄分散在每一步。前者是治理，后者是交互。

### MCP 扩展
Cline 支持 Model Context Protocol：
- 允许用户添加自定义工具
- 但工具添加本身需要人工确认

**关键限制**：MCP 扩展的是能力，不是责任。Cline 没有回答"新工具的责任由谁承担"。

### Checkpoints 系统
自动快照，支持对比和回滚：
- 技术层面的 undo
- 但无组织层面的"谁决定回滚"

---

## 4. 不可映射之处

- **无显式责任锚定**：人工确认是过程安全，不是责任归属
- **无契约复用**：每次任务都重新走确认流程，无法积累约束
- **无组织层**：假设单一用户与单一 agent，无多主体协作场景

---

## 5. 对 FAO 的有限启发

1. **IDE 作为约束界面**：diff 视图、快照系统证明界面设计可承载轻量治理
2. **Human-in-the-Loop 的边界**：适合高风险场景，但不适合高频、标准化的协作
3. **Checkpoints 的价值**：但 FAO 需要"谁有权回滚"的决策层，不只是技术层

---

## 6. 结论

Cline 证明了安全与效率的权衡可以有不同选择。但 FAO 的目标是让正常流转高效、异常场景可控——而不是默认所有场景都低效。Cline 是特定风险偏好下的解，不是通用框架。
