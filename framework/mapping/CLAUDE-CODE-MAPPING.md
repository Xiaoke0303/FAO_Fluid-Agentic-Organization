# Claude Code Mapping

> Claude Code 作为官方闭源 Agent 的映射参考。

---

## 1. 定位

**Claude Code 是厂商官方 runtime，不是开放框架。**

- Anthropic 官方出品，与 Claude 模型深度绑定
- 终端原生，强调代码库理解能力
- FAO 需要识别：官方闭源系统的治理黑箱问题

---

## 2. 与 FAO 核心差异

| 维度 | FAO 通用框架 | Claude Code |
|------|-------------|-------------|
| **开放性** | 开源/可审计 | 闭源/黑箱 |
| **模型绑定** | 模型无关 | 深度绑定 Claude |
| **治理可见性** | 文件化、可检查 | 内部实现不可见 |
| **责任锚定** | 显式契约 | 服务条款隐性约定 |
| **错误追溯** | 日志+写回 | 不可观测 |

---

## 3. 关键器官映射

### Codebase Understanding
Claude Code 的核心能力是**大规模代码库的理解**：
- AST 分析、符号检索、跨文件引用追踪
- 这是 runtime 能力，不是组织层能力

**与 FAO 差异**：FAO 不处理代码理解的技术问题，处理的是"谁可以授权修改代码"的治理问题。

### GitHub @claude 集成
允许在 GitHub 评论中 @claude 触发 agent：
- 这是一个跨平台触发点
- 但无显式 human checkpoint，agent 直接响应

**风险点**：公开仓库中任何人 @claude 都可能触发执行，责任边界模糊。

---

## 4. 不可映射之处

- **内部治理机制不可见**：如何决定 tool 调用、如何管理上下文、如何纠错——全部黑箱
- **责任归属隐性化**：用户通过服务条款间接接受责任，无 FAO 式的显式契约
- **无法定制约束**：用户只能接受 Anthropic 预设的安全层，无法像 FAO 那样自定义 ROLE-CONTRACT

---

## 5. 对 FAO 的有限启发

1. **代码库理解的标杆**：Claude Code 的 codebase 分析能力是 runtime 层的参考标准
2. **跨平台触发的便利性与风险**：@claude 证明跨平台触发有价值，但 FAO 需要配套的 human checkpoint
3. **闭源系统的治理真空**：正因为内部机制不可见，FAO 在外部建立责任锚定变得更必要

---

## 6. 结论

Claude Code 展示了官方闭源 Agent 的能力上限，也展示了其治理下限——内部机制不可审计、责任边界隐性化。FAO 的价值在对比中更清晰：为不可控的 runtime 提供外部的、显式的治理层。
